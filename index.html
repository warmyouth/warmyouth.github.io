initializer_list的实现

可以看到，initializer_list只有一个ctor，即initializer_list()。有两个问题：

  1. initializer_list如何转化类似{1,2,3,4}这样的初始化列表？
  2. private的ctor有什么用，“The compiler can call a private constructor”有什么含义？
    答案见[链接]](https://stackoverflow.com/questions/29200036/is-providing-a-private-constructor-for-initializer-list-conforming)。
    内有回答：

    This draft standard shows the synopsis for initializer_list.
    According to the synopsis the template only contains a default constructor, allowing you to create empty initializer_lists, which is clearly not very useful.
    However, initializer_list is a type that depends on some magic being done by the compiler.
    By magic, I’m referring to §8.5.4/5 that you’ve quoted. This allows the following statement to be legal and compile.
    std::initializer_list<int> numbers{1, 2, 3, 4}; // no such constructor in the synopsis
    Here, as explained in §8.5.4/5, the compiler will create an array containing the four integers,
    and then initialize the initializer_list instance with either a pair of pointers (first element and one past the end element) or a pointer and length (which is what both libstdc++ and libc++ seem to do).
    Once the instance has been created, your code is allowed to access all the member functions listed in the synopsis.
    The standard defines what you’d expect when constructing a std::initializer_list from an initializer list.
    The way it’s done is implementation-defined and, for what said above, it may therefore call a private constructor.

Indeed §17.6.5.5 (draft n° 3797) states:
A call to a member function signature described in the C ++ standard library behaves as if the implementation declares no additional member function signatures.

Footnote 189:
189) A valid C++ program always calls the expected library member function, or one with equivalent behavior. An implementation may also define additional member functions that would otherwise not be called by a valid C++ program.

即，为了实现initilizer_list，编译器做了一些额外的工作，单纯从initilizer_list的源码中是分析不出来的。
编译器在解析到initilizer_list变量时，会调用某些内置功能，将初始化列表赋值给该变量。

Posted on 2019-10-29

左值和右值

左值引用和右值引用的区别：

  1. 返回左值引用的函数，连同赋值、下标、解引用、前置递增/递减运算符，均生成左值，可以用来绑定左值引用。
  2. 返回非引用类型的函数，连同算术、关系、位、后置递增/递减运算符，均生成右值，可以用来绑定右值引用，或者const左值引用。
  3. 右值引用只能绑定到临时对象，临时对象没有其他用户使用，所以右值引用可以自由接管所引用对象的资源。

注意：
”单纯的变量“的表达式都是左值。即使一个变量a是右值引用类型（比如int &&a），a这个表达式也是左值，因此a不能绑定到右值引用类型上。
但可以通过std::move来获得“绑定到左值”的右值引用类型，只需保证不再使用移后源对象（之前的左值）。

注意：
移动ctor和移动=的定义比较困难，有两个原则：

  1. 移动ctor和移动=均不申请新内存，而是接管源对象管理的内存；
  2. 接管后需要设置源对象管理的内存为nullptr，以使其仍旧有效并且能够无感地销毁。

注意：
在模板实参推断时，有两条例外的绑定规则：

  1. 模板类型参数为T &&时，如果传递右值int，那么T被推断为int，如果传递左值int，那么T被推断为int &；
  2. 如果间接创建了引用的引用，那么这些引用发生了折叠，折叠规则为：
    • T& &，T& &&，T&& &均折叠成T &
    • T&& &&折叠成T &&

auto和decltype的区别

auto通过表达式的值推断待定义变量类型，decltype通过表达式的类型推断待定义变量的类型。

  1. decltype处理的表达式是单纯的变量时，会保留”顶层const“和”引用“，这点和auto相反；
  2. 处理不是单纯变量时，decltype返回表达式结果对应的类型，此时需要考虑结果的左值和右值的区分，左值时类型带引用，右值时类型不带引用。

这涉及到了C对于宏的一些规定。

由于STRINGIZE和__LINE__都是宏定义，STRINGIZE(__LINE__)就相当于嵌套宏。
嵌套宏类似于复合函数。复合函数的执行顺序是定义好的，程序语言和数学语言上表现一致，都是由内向外计算。
但嵌套宏的内外替换可以有先后顺序，C对嵌套宏的定义规则为：

  1. 总体而言，嵌套宏由内向外进行替换，即先展开宏参数（内部宏），再以递归的方式检查和替换已经展开的结果；
  2. 但如果在展开宏参数的过程中，发现某个宏参数遇到了#或##，那么就不再展开这个宏参数，而是直接应用#或##，没遇到#或##的宏参数仍旧按照规则1处理。

即，When macro arguments are substituted into the macro body, they are expanded unless they appear with the # or ## preprocessor operators in that macro.

Macro1例子：
f(g())中f是宏定义，但f定义中有#，所以不展开#对应的参数，即
f(g()) -> #g() -> "g()"
Macro2例子：
g(e(3))中g是宏定义，但g定义中有#，所以不展开#对应的参数，即
g(e(3)) -> #e(3), f(3, 0) -> "e(3)", 30
于是传入call得到call(“e(3)”, 30)。
h(e(3))中h是宏定义，但h定义中有##，所以不展开##对应的参数，即
h(e(3)) -> func_##e(3), f(3, 0) -> func_e(3), 30
于是传入call得到call(func_e(3), 30)。

hexo简介

Posted on 2019-10-25 Edited on 2019-10-26

简介

hexo是静态站点生成框架，基于Node.js。用户使用markdown语言写博客，写完后通过hexo将markdown转换成网页，并上传到服务器。因为生成的静态网页可以像代码一样在github上存储，所以一般的做法是结合hexo和github使用，可以省去搭建服务器的麻烦。
一般博客上还有一些诸如评论等类似“动态”的内容，这些内容如何在github上存储？这些可以通过插件实现。
比如评论可以使用Gitalk插件，其基于Github API开发，Gitalk把博客上的评论转换成对应issue上的问题；或者使用第三方服务实现，比如disqus。

安装

从官网下载node.js并解压，设置PATH。执行node -v和npm -v查看是否安装成功。
安装hexo：
npm install -g hexo-cli
升级hexo：
npm update -g hexo-cli

新建博客目录blog，进入blog目录并初始化：
hexo init
接着执行
npm install
安装必要的组件。

此时可以在本地验证hexo是否安装成功：

  1. 执行hexo g生成静态网页；
  2. 执行hexo s打开本地服务器；
  3. 浏览器打开http://localhost:4000/观看博客。

此外，hexo相关命令如下：

  1. 新建文章，hexo new "NewBlogName"；
  2. 生成静态网页，hexo generate；
  3. 启动服务器预览静态网页，hexo server；
  4. 推送静态网页，hexo deploy；
  5. 若发现网站有问题，可以考虑清除缓存，hexo clean。

连接github

接下来需要建立blog和github的联系。

申请github账号，配置ssh密钥，创建一个仓库，注意仓库名为用户名.github.io，并设置其为GitHub Pages。

在根目录下的_config.yml文件中，修改最后deploy项，修改为：
deploy:
type: git
repo: git@github.com:YourName/YourName.github.io.git
branch: master
其中YourName处填入github用户名。

安装和git相关的扩展：
npm i hexo-deployer-git

此时可以验证是否成功连接github：

  1. 执行hexo g生成静态网页；
  2. 执行hexo d将生成的静态网页推送到github上。

绑定域名

申请域名（比如mysite.com），添加解析CNAME，将某条主机记录（比如blogpage）设置为YourName.github.io，同样YourName为github用户名。

登录github，将之前创建的仓库的Settings中的Custom domain设置为域名blogpage.mysite.com。

在本地blog目录中，进入source子目录，创建名称为CNAME的文件，并填入域名blogpage.mysite.com。

接着hexo g和hexo d，并在浏览器中打开blogpage.mysite.com即可进入用户博客。

配置

修改根目录_config.yml中的站点信息，比如title，subtitle之类。
修改根目录_config.yml中的post_asset_folder选项，设置为false。这一块内容可以参考文档。
下载主题，本文使用next主题。
git clone https://github.com/theme-next/hexo-theme-next themes/next
通过git tag显示所有的版本号，通过git checkout切换到最新的稳定版本。
将根目录_config.yml中的theme选项改为next。

使用gitalk评论系统：
在github相关页面上注册新应用，其中Homepage URL和Authorization callback URL需要填使用gitalk的博客地址，即填域名blogpage.mysite.com。记录申请到的Client ID和Client Secret。
